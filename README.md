# azs-metrics — учебная аналитика АЗС

Проект для портфолио стажёра-аналитика. На примере данных АЗС считаю ключевые операционные метрики: реализация топлива, расхождения между счётчиками и накладными, остатки, доля расхождений по сменам. Проект показывает контроль качества данных, нормализацию справочников и прозрачную логику расчётов.

## Метрики, которые считаю

- `fuel_sold_liters` — объём реализованного топлива (по счётчикам ТРК).
- `discrepancy_liters` — расхождение между приходом по накладным и расходом по счётчикам.
- `discrepancy_rate` — доля расхождения относительно прихода (в %).
- `shift_avg_discrepancy` — среднее расхождение по сменам (чтобы видеть, где чаще ошибки).

## Откуда данные

Условные обезличенные данные в CSV:
- смены (shift_id, date, operator_name)
- приёмка топлива (delivery_date, volume_liters, fuel_type)
- показания счётчиков (date, meter_start, meter_end, fuel_type)

Данные обезличены, подходят для учебных задач.

## Как считаю (логика)

Основной SQL-запрос лежит в `query_azs_metrics.sql`. В нём:
- нормализую типы топлива («АИ‑92», «АИ92», «92» → «AI92»);
- считаю объём реализации как разницу между конечными и начальными показаниями счётчиков;
- свожу приход (накладные) и расход (счётчики) по дням и типам топлива;
- считаю расхождение и его долю;
- агрегирую по сменам и операторам для поиска аномалий.

## Результат

Дашборд в Yandex DataLens: карточки KPI, график расхождений по дням, таблица по сменам с фильтрами. Скриншоты — в папке `screenshots`.

## Зачем это в резюме

Проект показывает:
- работу с качеством данных (нормализация типов топлива, контроль дублей, проверка дат);
- расчёт операционных метрик и прозрачную логику (SQL + описание каждой метрики);
- визуализацию (BI-дашборд) и умение объяснять выводы простыми словами (педагогический навык);
- системность (README, понятная структура файлов, Issues как список задач).
